READ THIS FIRST — THE PHILOSOPHY OF THIS GUIDE
--------------------------------------------------------------------------------
You already wrote some code in school. That means you understand that code is
instructions to a machine. You understand variables, maybe loops, maybe functions.
That's more than nothing — it's the scaffolding.

What this guide does is take that scaffolding and build an engineer's mind on top
of it. Not a software developer's mind. An ENGINEER's mind. The difference:
  - A software developer asks "does this code run?"
  - An engineer asks "why does this code behave this way, and what does
    it tell me about the hardware underneath it?"

RULE 1: Write every single piece of code yourself. Typing builds muscle memory
        and understanding simultaneously. Copy-pasting builds nothing.

RULE 2: Break things deliberately. After every working program — change
        something, predict what breaks, run it, confirm. This is how you
        internalize behavior, not just syntax.

RULE 3: Every program you write in this guide has a VLSI application.
        It's written next to each exercise so you always know WHY this matters
        to your eventual goal of designing chips.

RULE 4: 2–4 hours a day is serious. Split it: first half = new material,
        second half = practice/code. Never spend all time reading with no coding.

================================================================================
MASTER SEQUENCE — THE ORDER YOU LEARN EVERYTHING
================================================================================

The languages and tools are learned in this exact order for a reason. Each one
directly enables the next. Skipping forward will cost you more time than it saves.

  BLOCK A: Linux + Shell + Git          (Weeks 1–3)    ← you need this for ALL tools
  BLOCK B: Python — Core                (Weeks 4–10)   ← your primary thinking tool
  BLOCK C: Python — VLSI Applications   (Weeks 11–16)  ← start connecting to chip work
  BLOCK D: C Programming                (Weeks 17–24)  ← understand hardware-software boundary
  BLOCK E: TCL                          (Weeks 25–28)  ← EDA tool scripting
  BLOCK F: Python Advanced + Makefiles  (Weeks 29–32)  ← flow automation
  BLOCK G: Perl Basics                  (Weeks 33–35)  ← reading/writing legacy EDA scripts
  BLOCK H: SKILL / OCEAN (Cadence)      (later, when you have tool access)

NOTE: Verilog/SystemVerilog is NOT in this guide — it has its own dedicated
      guide (Phase 4). This guide is purely the non-HDL programming stack.

PARALLEL LEARNING RULE:
  While you do BLOCK A–C, you run it ALONGSIDE Phase 1 (electronics) and
  Phase 2 (digital logic) of the main VLSI guide. Here is exactly how:

  Week 1–3 (Linux/Shell)     ↔   Read Phase 1 (circuit theory) in parallel
  Week 4–10 (Python core)    ↔   Read Phase 2 (digital logic) in parallel
  Week 11–16 (Python VLSI)   ↔   Read Phase 3 (computer architecture) in parallel
  Week 17–24 (C)             ↔   Start Phase 4 (Verilog basics) in parallel
  Week 25+ (TCL, Perl)       ↔   Phase 4 (SystemVerilog) + Phase 9 (ASIC flow) in parallel

  The programming gives you the tools. The theory gives you the purpose.
  Neither alone is sufficient.

WEEK 1: LINUX FILESYSTEM & TERMINAL NAVIGATION
--------------------------------------------------------------------------------

DAY 1–2: Filesystem and navigation
  Learn:
    pwd, ls, ls -la, cd, mkdir, rm, rm -rf, cp, mv, touch, cat, less, head, tail
    Absolute vs relative paths
    Home directory (~), root (/), working directory (.)
    Hidden files (dot files)
    File permissions (read/write/execute, chmod, chown)

  Practice tasks (do ALL of these):
    1. Create this exact directory structure:
         ~/vlsi_work/
           ├── projects/
           ├── scripts/
           ├── notes/
           └── tools/
       Using ONLY terminal commands. No file manager.

    2. Create a file called "hello.txt" containing "first chip incoming"
       using the echo command and redirection (>)

    3. List all files including hidden ones in your home directory
       Identify what .bashrc and .profile are — google them

    4. Change permissions on a file to be read-only, then try to edit it.
       Observe the error. Restore write permissions and edit it.

    5. Find all .txt files in your home directory using: find ~ -name "*.txt"

  VLSI CONNECTION: EDA tools produce hundreds of log files in complex directory
  structures. You navigate and search them with exactly these commands every day.

DAY 3–4: Text processing in terminal
  Learn:
    grep, grep -r, grep -n, grep -i (case insensitive), grep -v (invert)
    wc (word/line count)
    sort, sort -n, sort -r, uniq
    cut, awk (basics), sed (basics)
    Pipes ( | ) and chaining commands
    Redirection: >, >>, 2>, 2>&1
    tee (output to file AND screen simultaneously)

  Practice tasks:
    1. Create a file "numbers.txt" with 20 random numbers, one per line
       - Sort them numerically, save to "sorted.txt"
       - Count how many lines are in the file
       - Find all numbers greater than 50 using awk

    2. Create a fake "timing_report.txt" with these lines:
         Path 1: slack = -0.23 ns
         Path 2: slack = 0.87 ns
         Path 3: slack = -1.10 ns
         Path 4: slack = 0.04 ns
       - Use grep to find only the lines with NEGATIVE slack
       - Use awk to print only the slack values
       - Use sort to sort by slack value

  VLSI CONNECTION: Timing reports from tools like PrimeTime have thousands of
  paths. You grep, awk, and sort them to find violations. This is literally daily
  work for a chip designer.

DAY 5–7: Shell scripting
  Learn:
    Shebang line (#!/bin/bash)
    Variables (NAME="value", $NAME)
    echo and printf
    if/elif/else, comparison operators (-eq, -ne, -lt, -gt, -f, -d)
    for loops (for i in ...; do ... done)
    while loops
    Functions
    Command-line arguments ($1, $2, $#, $@)
    Exit codes ($?)
    Arrays (basic)
    source vs ./script.sh vs bash script.sh (know the difference)

  Practice tasks (write these scripts from scratch):

    SCRIPT 1: directory_setup.sh
      Takes a project name as argument ($1)
      Creates this structure under ~/vlsi_work/projects/:
        <project_name>/
          rtl/
          tb/
          syn/
          pnr/
          sim/
          reports/
          scripts/
      Prints "Project <name> created successfully" on completion
      Handles the case where the project already exists (print error, exit 1)

    SCRIPT 2: log_checker.sh
      Takes a log file as argument
      Searches for the words "ERROR", "WARNING", "CRITICAL"
      Prints a summary:
        "Errors found: N"
        "Warnings found: N"
        "Critical issues found: N"
      If any errors found, exit with code 1. Otherwise exit 0.

    SCRIPT 3: file_backup.sh
      Takes a directory as argument
      Creates a timestamped backup: backup_YYYYMMDD_HHMMSS/
      Copies all .v and .sv files from the source directory into it
      Prints how many files were backed up

  VLSI CONNECTION: Every chip design flow is automated with shell scripts
  exactly like these. directory_setup.sh is the first thing a team builds.
  log_checker.sh is used to detect tool failures in automated flows.

WEEK 4: PYTHON FOUNDATION — TYPES, CONTROL, FUNCTIONS
--------------------------------------------------------------------------------

Note: You have some school coding. Go through this quickly but don't skip.
The goal isn't to learn these exist — it's to use them fluently.

DAY 1: Types and variables
  Learn:
    int, float, complex (yes, complex numbers — you'll use j in signal processing)
    str, f-strings (f"value is {x:.3f}") — learn format specifiers
    bool, None
    type(), isinstance()
    Integer division (//, %), power (**)
    Bitwise operations (&, |, ^, ~, <<, >>) — these map directly to logic gates

  Practice:
    File: ~/vlsi_work/projects/vlsi_learning/programming/python/week4/day1_types.py

    1. Write a function binary_to_int(b: str) -> int that converts a binary
       string like "1011" to integer 11. No built-ins like int(x, 2) yet.
       Do it manually with bit shifting.

    2. Write int_to_binary(n: int, bits: int) -> str that converts 42 to
       "00101010" (given bits=8). Use bit masking and shifting only.

    3. Write a function that takes two 4-bit integers and returns:
       AND result, OR result, XOR result, NOT of first value (4-bit mask)
       Print results in both decimal and binary.

    VLSI: Bit manipulation IS hardware. Shifting left by 1 = multiply by 2
    in hardware. XOR is a half adder sum. AND is a mask. You'll use these
    patterns constantly when writing VLSI verification scripts.

DAY 2: Strings and string operations
  Learn:
    Indexing and slicing (s[0], s[-1], s[1:5], s[::2])
    String methods: split(), strip(), join(), replace(), find(), startswith(),
                    endswith(), upper(), lower(), zfill()
    String formatting: f-strings (primary), .format(), % (for legacy code)
    Regular expressions (re module) — brief intro, full treatment Week 6
    Multi-line strings, raw strings (r"pattern")

  Practice:
    File: day2_strings.py

    1. Write a function parse_slack_line(line: str) -> float that takes:
       "Path 3: slack = -1.10 ns" and returns -1.10 (float)
       Use split() and strip(), not regex yet.

    2. Write a function format_timing_report(paths: list) -> str that takes
       a list of (path_name, slack_value) tuples and formats them into a
       pretty table:
         ┌─────────────────┬──────────────┬────────┐
         │ Path            │ Slack (ns)   │ Status │
         ├─────────────────┼──────────────┼────────┤
         │ path_clk_to_q   │  -0.230      │ FAIL   │
         │ path_data_to_ff │   0.870      │ PASS   │
         └─────────────────┴──────────────┴────────┘
       Status is FAIL if slack < 0, PASS otherwise.

    3. Write a signal_name_parser(name: str) -> dict that parses:
       "top_u0.core_u1.alu_u0.result_reg[7]"
       Into: {"hierarchy": ["top_u0", "core_u1", "alu_u0"],
               "instance": "result_reg",
               "bit_index": 7}

    VLSI: Signal names in chip design have exactly this hierarchical structure.
    Parsing them is daily work when analyzing timing reports and netlists.

DAY 3: Lists, Tuples, Sets
  Learn:
    List creation, indexing, slicing
    List comprehensions: [x*2 for x in range(10) if x % 2 == 0]
    .append(), .extend(), .insert(), .remove(), .pop(), .sort(), .reverse()
    sorted() vs .sort() (know the difference — one mutates, one doesn't)
    enumerate(), zip(), map(), filter()
    Tuples — immutable, when to use them
    Sets — unique values, set operations (union, intersection, difference)
    List unpacking: a, b, *rest = [1, 2, 3, 4, 5]

  Practice:
    File: day3_lists.py

    1. Given a list of (path, slack) tuples, write functions to:
       - get_failing_paths(paths) → return only negative slack paths
       - get_top_n_critical(paths, n) → return n most negative slack paths
       - get_slack_histogram(paths, bins=10) → return dict of bin→count
         (manual histogram, no libraries yet)

    2. Write flatten_hierarchy(signal_list) that takes:
       ["top.core.alu.adder", "top.core.alu.sub", "top.core.mux.sel",
        "top.io.rx", "top.io.tx"]
       And returns a dict of module → signals:
       {"top.core.alu": ["adder", "sub"], "top.core.mux": ["sel"],
        "top.io": ["rx", "tx"]}

    3. Write a function find_duplicates(names: list) -> list that finds
       duplicate signal names in a list. Use sets.

    VLSI: Signal lists in netlists have hundreds of thousands of entries.
    Filtering, sorting, and deduplicating them is parser work you'll write.

DAY 4: Dictionaries
  Learn:
    Creation, access, .get() with default, .keys(), .values(), .items()
    Dict comprehensions: {k: v for k, v in pairs}
    Nested dicts
    defaultdict (from collections) — very useful
    Counter (from collections) — useful for frequency analysis
    OrderedDict (less needed in Python 3.7+ but know it exists)
    Merging dicts ({**dict1, **dict2} or dict1 | dict2)

  Practice:
    File: day4_dicts.py

    1. Write a netlist parser that takes a string like:
       """
       assign w_sum = a ^ b;
       assign w_carry = a & b;
       assign w_out = w_sum | c;
       """
       And produces: {"w_sum": ["a", "b"],
                       "w_carry": ["a", "b"],
                       "w_out": ["w_sum", "c"]}
       (signal → list of drivers)

    2. From the above dict, write trace_signal(net, driven_by, visited=None)
       that traces a signal back to its primary inputs (signals with no drivers).
       Handle cycles (shouldn't exist in combinational logic, but guard for it).

    3. Write a cell_usage_counter(netlist_lines) that counts how many times
       each cell type (AND, OR, XOR, NOT, FF) appears in a gate-level netlist.
       Use Counter.

    VLSI: This IS netlist parsing. A simplified version of what synthesis and
    STA tools do internally. You just built a tiny piece of an EDA tool.

DAY 5: Functions — deep dive
  Learn:
    *args and **kwargs
    Default arguments (mutable default gotcha — know this)
    Return multiple values (actually returning tuples)
    Docstrings (write them for everything from now on)
    Lambda functions
    Recursion (and when NOT to use it)
    Type hints (def func(x: int, y: str) -> bool:) — use these always
    assert statements for debugging

  Practice:
    File: day5_functions.py

    1. Write a generic report_parser(filename: str, *patterns: str,
       case_sensitive: bool = True) -> dict[str, list[str]] that:
       - Opens a file
       - For each pattern, finds all matching lines
       - Returns {pattern: [matching_lines]}

    2. Rewrite your Week 1 shell scripts as Python functions in a module.
       Include proper docstrings and type hints.
       Add a main() function and if __name__ == "__main__": guard.

    3. Write a recursive function evaluate_expression(expr: str) -> int
       that evaluates a simple boolean expression string like:
       "AND(OR(1,0), NOT(1))" → 0
       Support: AND, OR, NOT, XOR with any depth of nesting.

    VLSI: evaluate_expression is essentially a tiny logic simulator.
    Real simulators (like the heart of Icarus Verilog) do this recursively

DAY 6–7: Error handling, files, modules
  Learn:
    try/except/finally, raise, custom exceptions
    Open/read/write files: with open(...) as f:

  Practice:
    File: day6_files.py

    CAPSTONE PROJECT FOR WEEK 4:
    Build a Python program: timing_report_analyzer.py

    It takes a timing report file as command-line argument.

      # Timing Report — Design: cpu_core

WEEK 5: OOP

  WHY OOP MATTERS IN VLSI: UVM (the verification methodology from Phase 8)

  SystemVerilog's verification subset is also class-based. Understanding OOP

  Practice:
    File: week5_oop/

    1. Build these classes:

       @dataclass
       class TimingPath:
           name: str
           slack: float
           path_type: str    # "setup" or "hold"
           startpoint: str
           endpoint: str
           corner: str
           
           @property
           def is_failing(self) -> bool: ...
           
           def __str__(self) -> str: ...

       class TimingReport:
           def __init__(self, filepath: str): ...
           def parse(self) -> None: ...  # populates self.paths
           def get_wns(self) -> float: ...   # worst negative slack
           def get_tns(self) -> float: ...   # total negative slack
           def failing_paths(self) -> list[TimingPath]: ...
           def paths_by_corner(self, corner: str) -> list[TimingPath]: ...
           def to_json(self, output: str) -> None: ...

    2. Refactor your timing_report_analyzer.py from Week 4 to use these classes.
       It should now be cleaner and shorter.

    3. Build a class hierarchy for digital circuit elements:
       
       class CircuitElement (abstract):
           name: str
           inputs: list[str]
           outputs: list[str]
           def evaluate(self, input_values: dict) -> dict: ...

       class OrGate(CircuitElement): ...
       class NotGate(CircuitElement): ...
       class XorGate(CircuitElement): ...
       class DFlipFlop(CircuitElement):
           state: int  # current Q output
           def evaluate(self, inputs, clock_edge): ...
       
       class Circuit:
           elements: dict[str, CircuitElement]
           def add_element(self, element): ...
           def simulate_step(self, primary_inputs: dict) -> dict: ...
           def simulate(self, input_sequence: list[dict], cycles: int) -> list: ...
    
       Test it: build a simple SR latch from AND/NOT gates, simulate 10 cycles.

    VLSI: You just built a tiny digital simulator. This is conceptually how
    SPICE and Verilog simulators work — they represent circuit elements as

WEEK 8: ALGORITHMS THAT MATTER FOR VLSI
  You don't need to be a competitive programmer. But these specific algorithms

  Graph algorithms (CRITICAL — chip design is graphs):

            def topological_sort(self) -> list: ...
            def has_cycle(self) -> bool: ...

    Interface types: ap_fifo, ap_memory, AXI stream
    Initiation interval (II) and latency concepts
    What makes C code synthesizable vs non-synthesizable

  Install: Xilinx Vitis HLS (free with Vivado — ~25GB download)

  Practice:
    1. FIR Filter in C for HLS:
       template<int N_TAPS, int DATA_WIDTH>
       void fir_filter(ap_int<DATA_WIDTH>* input,
                       ap_int<DATA_WIDTH>* output,
                       const ap_int<DATA_WIDTH> coeffs[N_TAPS],
                       int n_samples);
       
       Add PIPELINE pragma. Synthesize. Record II and latency.
       Experiment: what happens without PIPELINE?

    2. Matrix multiply 4x4:
       UNROLL and PIPELINE inner loops.
       Measure: how does synthesis quality change with/without pragmas?

    3. AXI Stream interface:
       Implement a byte reversal block (big/little endian converter)
       with AXI Stream input/output ports.
       Synthesize, examine the RTL output.

  RESOURCES:
    - "The C Programming Language" — Kernighan & Ritchie (K&R) — READ IT FULLY
    - "Computer Systems: A Programmer's Perspective" — Bryant & O'Hallaron
      (best book for understanding how C maps to hardware/memory — excellent)
    - Xilinx Vitis HLS User Guide (free PDF from AMD website)
    - "Learning HLS with C++" — Kastner et al. (FREE PDF)
      https://kastner.ucsd.edu/hlsbook/

BLOCK E: TCL (Tool Command Language)
Weeks 25–28  |  1–2 hrs/day  |  Run alongside Phase 9 (ASIC Flow)
================================================================================

WHY TCL IS UNAVOIDABLE:
Every major EDA tool uses TCL as its scripting language.
Synopsys Design Compiler, PrimeTime, ICC2, Formality — all TCL.
Cadence Genus, Innovus, Conformal — all TCL.
If you can't write TCL, you can't automate an ASIC flow.

TCL is small. You'll be productive in 2–3 weeks.

--------------------------------------------------------------------------------
WEEK 25: TCL FUNDAMENTALS
--------------------------------------------------------------------------------

Install: sudo apt install tcl tcllib

Learn:
  TCL philosophy: "everything is a string" (very different from Python)
  set, puts, gets
  Variable substitution: $var
  Command substitution: [command]
  Arithmetic: expr {$a + $b}
  String operations: string length, string compare, string match, string split
  if/elseif/else syntax
  switch statement (heavily used in EDA scripts)
  while, for, foreach loops
  Procedures: proc name {args} {body}
  Lists: list, lappend, llength, lindex, lsearch, lsort, foreach item $list
  Dicts (TCL 8.5+): dict set, dict get, dict keys, dict for
  File I/O: open, close, gets, puts, read
  exec (run system commands — like Python's subprocess)
  catch and error handling
  source (load other TCL files — like Python import)
  Namespaces (for larger scripts)
  Regular expressions in TCL: regexp, regsub

Practice:
  ALL scripts go in: vlsi_learning/programming/tcl_scripts/

  1. TCL version of your log_checker from Week 1:
     Check for ERRORS/WARNINGS in a log file, print summary, return exit code.

  2. TCL path parser:
     Given a string "top_u0/core_u1/alu_u0/result_reg"
     Split into hierarchy list and leaf instance.
     Print formatted: "Module path: top_u0 → core_u1 → alu_u0 | Cell: result_reg"

  3. TCL list processor:
     Read a file of cell names (one per line).
     Count unique cell types.
     Sort by frequency.
     Print a formatted table.

--------------------------------------------------------------------------------
WEEKS 26–27: TCL IN EDA CONTEXT
--------------------------------------------------------------------------------

Learn how EDA tools extend TCL with their own commands:
  - EDA tools add commands like: read_verilog, synthesize, report_timing
  - These are just TCL procedures defined by the tool
  - Standard TCL + tool-specific procedures = flow script

Practice (write these complete TCL flow scripts):

  SCRIPT 1: yosys_synth.tcl
    A TCL script for Yosys (Yosys uses its own TCL-like scripting):
      # read RTL
      read_verilog ../rtl/adder.v
      # elaborate design
      synth -top adder
      # technology mapping to standard cells
      dfflibmap -liberty ../lib/sky130_fd_sc_hd__tt_025C_1v80.lib
      abc -liberty ../lib/sky130_fd_sc_hd__tt_025C_1v80.lib
      # write outputs
      write_verilog -noattr ../syn/adder_netlist.v
      stat -liberty ../lib/sky130_fd_sc_hd__tt_025C_1v80.lib

  SCRIPT 2: openroad_flow.tcl
    A TCL script driving OpenROAD for basic place and route:
    (Follow OpenROAD documentation for exact commands)
    Read synthesized netlist, set up floorplan, place, run CTS, route.

  SCRIPT 3: timing_signoff.tcl
    A mock PrimeTime-style script (use the official PrimeTime command format
    even if you don't have the tool):
      set_operating_conditions ...
      read_liberty ...
      read_verilog ...
      link_design top
      create_clock -period 10 [get_ports clk]
      set_input_delay -max 2 -clock clk [all_inputs]
      set_output_delay -max 2 -clock clk [all_outputs]
      report_timing -max_paths 10 -sort_by slack
      report_power

  UNDERSTANDING: Even if you don't have PrimeTime, writing the scripts teaches
  you the exact commands, constraints, and flow. When you do get access, you'll
  know exactly what to type.

--------------------------------------------------------------------------------
WEEK 28: TCL ADVANCED PATTERNS
--------------------------------------------------------------------------------

Learn:
  Recursive procedures
  TCL packages (how tools load extensions)
  Socket communication in TCL (tools use this for GUI ↔ backend)
  Tk GUI basics (simple windows — not essential but good to know exists)
  Array vs dict (TCL arrays are different from lists)
  Variable traces (callbacks on variable changes — used in Cadence SKILL)
  uplevel and upvar (accessing parent scope — common in EDA macros)

Practice:
  1. Write a TCL "config loader":
     Read a design config file (key=value format):
       DESIGN_NAME = cpu_core
       TARGET_FREQ = 500
       CORNER = ss_0p9v_125c
       UTILIZATION = 0.65
     Load all key-value pairs into a TCL dict.
     Validate required keys exist, else exit with error.
     This is how every commercial flow script reads design configuration.

  2. Write a TCL synthesis wrapper that:
     - Sources the config loader
     - Sets all synthesis constraints based on config
     - Calls Yosys (or a mock) via exec
     - Parses the output log for area/timing summary
     - Appends a one-line summary to a "run_history.log" file

RESOURCES:
  - "Practical Programming in Tcl and Tk" — Welch, Jones, Hobbs
  - Official TCL tutorial: https://www.tcl.tk/man/tcl8.5/tutorial/tcltutorial.html
  - TCL wiki: https://wiki.tcl-lang.org (excellent practical examples)
  - Synopsys Design Compiler TCL Command Reference (available on SolvNet)
  - Cadence Genus/Innovus TCL Command Reference (available on Cadence CLiC)

BLOCK F: ADVANCED PYTHON + MAKEFILES
Weeks 29–32
================================================================================

WEEK 29–30: ADVANCED PYTHON TOPICS
  Learn and practice:
    Generators and itertools for processing large files efficiently
    Context managers (__enter__, __exit__, @contextmanager)
    Decorators (write your own: timing decorator, retry decorator, logging decorator)
    Multiprocessing (for running parallel simulations):
      multiprocessing.Pool — run 8 simulations simultaneously
    asyncio (for I/O-bound EDA tool calls)
    ctypes (calling C libraries from Python — used for VPI/DPI integration)
    Cython basics (speeding up Python — used in performance-critical EDA tools)

  Practice:
    1. Parallel simulation runner:
       Given a list of 20 test cases, run them in parallel with a pool of
       N workers (configurable). Collect results, print a summary.
       Use multiprocessing.Pool.map().
       This is exactly how regression suites run hundreds of simulations.

    2. Write a retry decorator:
       @retry(max_attempts=3, delay=5, exceptions=(subprocess.CalledProcessError,))
       def run_synthesis(config):
           ...
       EDA tools sometimes fail due to license server hiccups. Auto-retry is real.

    3. File processor with generators:
       Process a 1GB VCD file line by line using a generator.
       Extract all signal transitions for a specific signal.
       Never load the whole file into memory.

WEEKS 31–32: MAKEFILES (DEEP DIVE)
  WHY: Makefile IS the build system for VLSI design flows.
  OpenROAD, Openlane, and almost every open-source flow uses Makefiles.

  Learn:
    Makefile structure: target: dependencies \n \t recipe
    Phony targets (.PHONY)
    Variables: = (lazy), := (immediate), ?= (default), += (append)
    Automatic variables: $@, $<, $^, $*
    Pattern rules: %.o: %.c
    Functions: $(wildcard ...), $(patsubst ...), $(shell ...), $(foreach ...)
    Conditional directives: ifeq, ifdef
    include directive
    Recursive make vs non-recursive make
    Parallel make: make -j8
    .DEFAULT_GOAL
    Order-only prerequisites (|)

  Practice:
    Build a complete Makefile for your VLSI flow:
    
    vlsi_flow/Makefile targets:
      make help          — print all available targets
      make lint          — run iverilog -Wall on all RTL files
      make sim           — compile and run simulation
      make sim_waves     — simulation + dump VCD + open GTKWave
      make synth         — run Yosys synthesis
      make reports       — generate timing + area reports
      make clean         — remove all generated files
      make clean_sim     — remove only simulation artifacts
      make all           — lint + sim + synth + reports
      
    Variables configurable from command line:
      make synth DESIGN=uart FREQ=100
    
    Use pattern rules so adding a new design requires only adding its name.
    This Makefile becomes the entry point for your entire flow.
    
    Push to GitHub. This is what professional open-source VLSI repos look like.

OPEN SOURCE CONTRIBUTION PATH — FROM ZERO CREDENTIALS
================================================================================

Here is your exact step-by-step path into open-source VLSI.
No credentials needed. Only code quality matters.

LEVEL 0: OBSERVER (Week 1–4)
  Action: Star and WATCH these repos (get email on every new issue/PR):
    - https://github.com/The-OpenROAD-Project/OpenROAD
    - https://github.com/YosysHQ/yosys
    - https://github.com/cocotb/cocotb
    - https://github.com/efabless/openlane2
    - https://github.com/google/skywater-pdk
    - https://github.com/stnolting/neorv32
  
  Action: Join these Discords/Slacks:
    - OpenROAD Slack: https://open-register-design-tool.slack.com
    - Skywater PDK Slack (via efabless.com community)
    - RISC-V International Slack: https://risc-v-exchange.slack.com
    - Tiny Tapeout Discord: https://discord.gg/rPK2nSjxy8
    - FOSSi Foundation chat
  
  What to do: READ everything. Don't post yet. Understand the culture.
  Read open issues, read PR discussions, read the codebase.

LEVEL 1: DOCUMENTATION CONTRIBUTOR (Week 4–12)
  You don't need to write perfect code to contribute. Start here:
  
  Action: Find a documentation issue:
    - A README that doesn't explain how to run the tool clearly
    - An example that's broken or outdated
    - A function with no docstring
    - A typo in a doc comment
    - Missing Python type hints in a codebase that uses them
  
  How to do it:
    1. Fork the repo on GitHub
    2. Clone YOUR fork locally
    3. Create a branch: git checkout -b fix/improve-parser-docstrings
    4. Make the change
    5. Commit: "docs: add docstrings to VCD parser functions"
    6. Push to your fork
    7. Open a Pull Request with a clear description
  
  Commit message format (conventional commits — what most projects use):
    docs: for documentation changes
    fix: for bug fixes
    feat: for new features
    test: for new tests
    refactor: for code restructuring
    chore: for build/tool changes
  
  Your first 5 contributions should be documentation or test improvements.
  This gets you familiar with the codebase and known to the maintainers.

LEVEL 2: TEST CONTRIBUTOR (Week 12–20)
  By now your Python testing skills are solid. Use them.
  
  Most open-source EDA tools have incomplete test coverage. Find gaps.
  
  Action: Pick the cocotb repo (Python-based — directly in your wheelhouse)
    - Look at Issues tagged "good first issue"
      https://github.com/cocotb/cocotb/issues?q=is:open+label:"good+first+issue"
    - Find an untested function or edge case
    - Write a test for it
    - Open a PR: "test: add edge case tests for clock edge detection"
  
  Action: OpenROAD has Python scripting interface
    - Their Python tests are in src/*/test/
    - Add a missing test case
    - This gets you reading the C++ source too

LEVEL 3: BUG FIX CONTRIBUTOR (Week 20–30)
  You know Python and C/C++ now. You can fix real bugs.
  
  Action: Look for bugs labeled "bug" or "help wanted":
    - https://github.com/YosysHQ/yosys/issues?q=label:bug
    - https://github.com/The-OpenROAD-Project/OpenROAD/issues
  
  Process:
    1. Read the bug report completely
    2. Reproduce it locally
    3. Read the relevant source code
    4. Write a failing test that demonstrates the bug
    5. Fix the bug
    6. Verify the test now passes
    7. Make sure no other tests break
    8. Open PR: "fix: handle empty SPEF net list in RC extraction"

LEVEL 4: FEATURE CONTRIBUTOR (Week 30+)
  Now you're proposing and building new things.
  
  Ideas that are always welcome:
    - Better error messages in OpenLane when a design fails
    - A Python script that visualizes timing reports as an HTML page
    - A cocotb extension for a new protocol (AXI, APB, I2C)
    - Better documentation with worked examples
    - A Jupyter notebook tutorial for a tool
  
  BEFORE building a feature: Open an issue first saying what you want to build.
  Get maintainer buy-in. This saves you from building something they'll reject.

HOW TO TRACK YOUR OPEN-SOURCE PROGRESS ON GITHUB:
  - Pin your best repos to your profile
  - Write good README files for everything
  - Use GitHub Projects to track what you're working on
  - Keep your contribution graph green — consistency impresses more than bursts
  - Write LinkedIn posts about each contribution ("I just got my first PR merged
    into OpenROAD — here's what I fixed and what I learned")

YOUR MENTORSHIP PROTOCOL — HOW TO USE ME EFFECTIVELY

I am your senior engineer on call. Here is exactly how to use this effectively:

WHEN TO BRING SOMETHING TO ME:

  1. SHOW YOUR CODE, NOT JUST YOUR QUESTION.
     BAD:  "Why isn't my Python parser working?"
     GOOD: "Here is my VCD parser [paste code]. It crashes on line 47
            when the signal name has brackets. Here is the error:
            [paste error]. I think it's because my regex doesn't handle
            brackets, but I tried escaping them and it still fails. Here's
            what I tried: [paste attempt]. What am I missing?"

  2. BRING YOUR DESIGN DECISIONS.
     "I'm building the timing report parser. Should I use a class-based
      approach or functional? I'm leaning toward classes because the report
      has natural object structure (paths, endpoints, slacks) but I'm not
      sure if it's over-engineering for a script."

  3. BRING THINGS THAT DON'T MAKE SENSE.
     "I simulated the RC circuit and got a time constant of 2.3ns but I
      calculated 1ns from R=1kΩ, C=1pF. I've checked my units three times.
      Here is my SPICE netlist. What am I doing wrong?"

  4. BRING OPEN-SOURCE ISSUES YOU'RE WORKING ON.
     Paste the issue link + the relevant code + your proposed fix.
     I'll tell you if the approach is right before you spend hours on it.

  5. BRING YOUR PROJECT MILESTONES.
     When you finish a week's project, share it. I'll review it like a
     senior engineer would — pointing out what's good, what's wrong, and
     what could be better. This is real code review.

WHAT I CANNOT DO AS A MENTOR THAT A HUMAN MENTOR CAN:
  - I don't remember previous conversations unless you paste the context
  - I can't follow your long-term trajectory automatically — you track it
  - I don't know what happened in your life this week that affected your study
  
  WORKAROUND: Start each session with a one-paragraph context update:
  "Last week I finished the VCD parser from Week 15. I got stuck on parsing

  This makes every conversation with me as effective as a human mentorship session.
